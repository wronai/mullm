# projekt nazywa sie mullm, kontynuuj

Jasne — kontynuując dla projektu **mullm**, proponuję potraktować go jako event-sourced control plane dla multi-agent runtime, z prostym MVP opartym o EventStoreDB, NATS, Postgres, orchestrator, projector, web UI i workerów shellowych. EventStoreDB da Ci trwały log zdarzeń i Admin UI na porcie 2113 w trybie single-node do lokalnych prób, a NATS dobrze nadaje się do lekkiej komunikacji między usługami i workerami w Compose.[^1][^2][^3]

## Domena mullm

Dla mullm zacząłbym od małego, spójnego jądra domenowego zamiast pełnej eksplozji bytów. W event sourcing najlepiej sprawdzają się agregaty o wyraźnych granicach odpowiedzialności, a źródła CQRS/ES podkreślają, że komendy reprezentują intencję, a eventy fakty dokonane.[^4][^5][^6]

Minimalne agregaty dla mullm:

- `TaskAggregate` — jednostka pracy.
- `FlowAggregate` — proces obejmujący wiele tasków.
- `AgentAggregate` — agent z capability i stanem.
- `WorkflowAggregate` — wersja definicji procesu.
- `PluginAggregate` — instalacja i aktywacja pluginów.
- `ApprovalAggregate` — bramka decyzyjna dla ryzykownych działań.[^7][^8][^9]


## Komendy i eventy

Na MVP dla mullm wystarczy skupić się na taskach, agentach i workflowach, a resztę dodawać stopniowo. Wzorzec „command → aggregate → events → projections” jest podstawą klasycznych implementacji CQRS/ES.[^5][^10][^11]

Przykładowy zestaw:


| Agregat | Komendy | Eventy |
| :-- | :-- | :-- |
| `TaskAggregate` | `CreateTask`, `DelegateTask`, `ClaimTask`, `RaisePriority`, `LowerPriority`, `StartTaskExecution`, `PauseTask`, `ResumeTask`, `CompleteTask`, `FailTask`, `SwitchExecutionMode` [^10][^4] | `TaskCreated`, `TaskDelegated`, `TaskClaimed`, `TaskPriorityRaised`, `TaskPriorityLowered`, `TaskExecutionStarted`, `TaskPaused`, `TaskResumed`, `TaskCompleted`, `TaskFailed`, `ExecutionModeSwitched` [^7][^8] |
| `AgentAggregate` | `RegisterAgent`, `HeartbeatAgent`, `AssignTaskToAgent`, `MarkAgentBusy`, `MarkAgentIdle`, `UpdateAgentCapabilities`, `DisableAgent`, `EnableAgent` [^12][^8] | `AgentRegistered`, `AgentHeartbeatReceived`, `TaskAssignedToAgent`, `AgentMarkedBusy`, `AgentMarkedIdle`, `AgentCapabilitiesUpdated`, `AgentDisabled`, `AgentEnabled` [^12][^8] |
| `WorkflowAggregate` | `ProposeWorkflowVersion`, `ValidateWorkflowVersion`, `ApproveWorkflowVersion`, `ActivateWorkflowVersion`, `RollbackWorkflowVersion` [^8][^9] | `WorkflowVersionProposed`, `WorkflowVersionValidated`, `WorkflowVersionApproved`, `WorkflowVersionActivated`, `WorkflowVersionRolledBack` [^8][^9] |
| `PluginAggregate` | `ProposePlugin`, `ValidatePlugin`, `InstallPlugin`, `ActivatePlugin`, `RollbackPlugin` [^9] | `PluginProposed`, `PluginValidated`, `PluginInstalled`, `PluginActivated`, `PluginRolledBack` [^9] |
| `ApprovalAggregate` | `CreateApprovalRequest`, `ApproveRequest`, `RejectRequest`, `ExpireApproval` [^8][^9] | `ApprovalRequested`, `ApprovalGranted`, `ApprovalRejected`, `ApprovalExpired` [^8][^9] |

## Read modele

Dla mullm UI powinno czytać z projections w Postgresie, nie bezpośrednio z event store. To dokładnie odpowiada idei CQRS: osobny model zapisu i osobny model odczytu zoptymalizowany pod konkretne widoki.[^13][^7]

Na start zrób 4 projections:

- `operational_feed` — feed wszystkiego, co dzieje się w systemie.[^8][^12]
- `task_board` — zadania, owner, agent, priorytet, tryb.[^7]
- `agent_fleet` — agenci, capability, obciążenie, health.[^12]
- `workflow_versions` — aktywne i oczekujące definicje workflowów.[^9]


## JSON komend i eventów

Najlepiej od razu ustalić prosty envelope dla komend i eventów. Dzięki temu orchestrator, UI i workery będą mówiły tym samym językiem, a logika routingu będzie prostsza.[^11][^5]

Przykład komendy:

```json
{
  "command_id": "cmd-01J...",
  "aggregate_type": "Task",
  "aggregate_id": "task-123",
  "command_type": "CreateTask",
  "occurred_at": "2026-06-04T15:30:00Z",
  "actor": {
    "type": "user",
    "id": "user-admin"
  },
  "payload": {
    "title": "Sprawdź pipeline staging",
    "priority": 50,
    "execution_mode": "semi_auto",
    "required_capabilities": ["shell", "browser"],
    "workflow_id": "wf-deploy-review"
  }
}
```

To jest zgodne z praktyką, że komenda reprezentuje intencję użytkownika lub systemu.[^4][^5]

Przykład eventu:

```json
{
  "event_id": "evt-01J...",
  "stream_id": "Task-task-123",
  "aggregate_type": "Task",
  "aggregate_id": "task-123",
  "event_type": "TaskCreated",
  "revision": 1,
  "occurred_at": "2026-06-04T15:30:01Z",
  "causation_id": "cmd-01J...",
  "correlation_id": "flow-456",
  "payload": {
    "title": "Sprawdź pipeline staging",
    "priority": 50,
    "execution_mode": "semi_auto",
    "status": "created"
  },
  "metadata": {
    "tenant_id": "acme-software",
    "environment": "dev"
  }
}
```

Taki wzorzec dobrze wspiera projections, audit i replay.[^5][^8][^7]

## Stack MVP w Compose

Dla mullm MVP zrób w jednym `docker-compose.yml`, bo przy lokalnych próbach i testach jednej sieci to najprostsze rozwiązanie. NATS można bez problemu użyć jako wspólnego brokera w jednym Compose, a EventStoreDB może działać jako single-node insecure do developmentu z dostępem przez `esdb://localhost:2113?tls=false`.[^2][^14][^3]

Proponowany stack:

- `eventstoredb` — event store.[^2]
- `nats` — broker dla runtime/pubsub.[^3]
- `postgres` — read models.[^15]
- `mullm-orchestrator` — command handlers, sagi, router.
- `mullm-projector` — aktualizacja projections.
- `mullm-web` — panel.
- `mullm-shell-agent-a`
- `mullm-shell-agent-b`[^3][^2]


## Szkic Compose

Najprostszy szkic dla mullm wyglądałby tak:

```yaml
services:
  eventstoredb:
    image: eventstore/eventstore:latest
    container_name: mullm-eventstoredb
    command: >
      --insecure
      --run-projections=All
      --enable-atom-pub-over-http
    ports:
      - "2113:2113"
    volumes:
      - eventstore-data:/var/lib/eventstore

  nats:
    image: nats:latest
    container_name: mullm-nats
    command: ["--http_port", "8222"]
    ports:
      - "4222:4222"
      - "8222:8222"

  postgres:
    image: postgres:16
    container_name: mullm-postgres
    environment:
      POSTGRES_DB: mullm
      POSTGRES_USER: mullm
      POSTGRES_PASSWORD: mullm
    ports:
      - "5432:5432"
    volumes:
      - postgres-data:/var/lib/postgresql/data

  orchestrator:
    build: ./services/orchestrator
    container_name: mullm-orchestrator
    depends_on:
      - eventstoredb
      - nats
      - postgres
    environment:
      EVENTSTORE_URL: esdb://eventstoredb:2113?tls=false
      NATS_URL: nats://nats:4222
      DATABASE_URL: postgres://mullm:mullm@postgres:5432/mullm

  projector:
    build: ./services/projector
    container_name: mullm-projector
    depends_on:
      - eventstoredb
      - postgres
    environment:
      EVENTSTORE_URL: esdb://eventstoredb:2113?tls=false
      DATABASE_URL: postgres://mullm:mullm@postgres:5432/mullm

  web:
    build: ./services/web
    container_name: mullm-web
    depends_on:
      - orchestrator
      - postgres
    ports:
      - "3000:3000"

  shell-agent-a:
    build: ./agents/shell-agent
    container_name: mullm-shell-agent-a
    depends_on:
      - nats
    environment:
      AGENT_ID: shell-agent-a
      NATS_URL: nats://nats:4222

  shell-agent-b:
    build: ./agents/shell-agent
    container_name: mullm-shell-agent-b
    depends_on:
      - nats
    environment:
      AGENT_ID: shell-agent-b
      NATS_URL: nats://nats:4222

volumes:
  eventstore-data:
  postgres-data:
```

Ten układ jest sensowny dla MVP, bo bazuje na wspieranym wzorcu single-node EventStoreDB do lokalnych testów i prostym NATS uruchamianym w Dockerze.[^1][^2][^3]

## Przepływ w mullm

W mullm przepływ powinien być taki:

1. Web UI wysyła komendę do orchestratora.[^10]
2. Orchestrator ładuje stan agregatu z EventStoreDB i zapisuje nowe eventy.[^2][^5]
3. Projector buduje read model w Postgresie.[^15][^7]
4. Orchestrator lub saga publikuje runtime task do NATS.[^3]
5. Shell agent wykonuje zadanie i odsyła wynik jako nową komendę albo event wejściowy do write side.[^16][^17]

## Co dalej w mullm

Najbliższy sensowny krok to dopisać:

- `TaskAggregate`, `AgentAggregate`, `WorkflowAggregate`,
- tabelki projections w Postgresie,
- jeden endpoint `POST /commands`,
- i jeden shell worker subskrybujący `task.assigned.shell`.[^10][^5][^3]

To da Ci pierwszy działający rdzeń mullm: event store, CQRS, projections, routing i dwóch agentów wykonawczych.[^5][^2][^3]

Mogę teraz pójść krok dalej i przygotować:

1. **konkretną strukturę repo mullm**,
2. **SQL dla projections**,
3. albo **przykładowe pliki startowe usług orchestrator/projector/shell-agent**.
