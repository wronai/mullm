# lista eventów/komend/agregatów CQRS/ES,

praktyczny stack MVP w Docker Compose pod taki system.

Tak — dla takiego systemu warto rozpisać model domeny od **agregatów, komend i eventów**, a MVP w Compose oprzeć o prosty stack: event store + bus + query DB + orchestrator + workers + web UI. EventStoreDB jest naturalnym kandydatem do event-native persistence, a Docker Compose dobrze nadaje się do lokalnego uruchomienia pojedynczego węzła oraz całego środowiska testowego.[^1][^2][^3][^4]

## Model CQRS/ES

Najprostszy sensowny model domeny dla Twojego systemu to kilka agregatów zamiast jednego „super bytu”. W event sourcing agregat powinien chronić spójność konkretnej granicy biznesowej, a nie obejmować całego świata zadań, agentów i pluginów naraz.[^5][^6][^1]

Proponowane agregaty:

- **TaskAggregate** — pojedyncze zadanie operacyjne.
- **FlowAggregate** — większy proces złożony z wielu tasków.
- **AgentAggregate** — stan i capability agenta.
- **MachineAggregate** — host, sesje, środowiska, health.
- **PluginAggregate** — lifecycle pluginu/usługi.
- **WorkflowAggregate** — wersjonowana definicja workflow.
- **InstallationAggregate** — bootstrap i instalacja node agenta.
- **ApprovalAggregate** — approval gate dla zmian i akcji ryzykownych.[^7][^8][^9]


## Komendy

Komendy powinny być jawne, imperatywne i walidowane przez agregat, a ich wynik to jeden lub więcej eventów. W CQRS to komendy są wejściem do modelu zapisu, podczas gdy UI czy automation nie powinny mutować stanu bezpośrednio.[^10][^1]

Przykładowe komendy per agregat:


| Agregat | Komendy |
| :-- | :-- |
| TaskAggregate | `CreateTask`, `DelegateTask`, `ClaimTask`, `RaisePriority`, `LowerPriority`, `StartTaskExecution`, `PauseTask`, `ResumeTask`, `CompleteTask`, `FailTask`, `RequestHumanTakeover`, `SwitchExecutionMode` [^1][^10] |
| FlowAggregate | `CreateFlow`, `AttachTaskToFlow`, `StartFlow`, `PauseFlow`, `ResumeFlow`, `CancelFlow`, `CloseFlow` [^7][^8] |
| AgentAggregate | `RegisterAgent`, `HeartbeatAgent`, `AssignTaskToAgent`, `MarkAgentBusy`, `MarkAgentIdle`, `DisableAgent`, `EnableAgent`, `UpdateAgentCapabilities` [^11][^8] |
| MachineAggregate | `RegisterMachine`, `ReportMachineHealth`, `OpenSession`, `CloseSession`, `AttachBrowserCapability`, `AttachShellCapability` [^11][^8] |
| PluginAggregate | `ProposePlugin`, `BuildPlugin`, `ValidatePlugin`, `InstallPlugin`, `ActivatePlugin`, `DisablePlugin`, `RollbackPlugin` [^12][^9] |
| WorkflowAggregate | `ProposeWorkflowVersion`, `ValidateWorkflowVersion`, `ApproveWorkflowVersion`, `ActivateWorkflowVersion`, `RollbackWorkflowVersion` [^8][^9] |
| InstallationAggregate | `IssueInstallToken`, `RegisterInstallation`, `ConfirmInstallation`, `UpgradeInstallation` [^3][^4] |
| ApprovalAggregate | `CreateApprovalRequest`, `ApproveRequest`, `RejectRequest`, `ExpireApproval` [^9][^8] |

## Eventy

Eventy opisują to, co już zaszło, i stają się trwałym logiem audytowym oraz źródłem projections. Event-sourced systemy agentowe szczególnie korzystają z pełnej historii, bo można potem odtwarzać decyzje, budować read modele i ewaluować skuteczność workflowów.[^8][^7]

Przykładowe eventy:


| Agregat | Eventy |
| :-- | :-- |
| TaskAggregate | `TaskCreated`, `TaskDelegated`, `TaskClaimed`, `TaskPriorityRaised`, `TaskPriorityLowered`, `TaskExecutionStarted`, `TaskPaused`, `TaskResumed`, `TaskCompleted`, `TaskFailed`, `HumanTakeoverRequested`, `ExecutionModeSwitched` [^7][^8] |
| FlowAggregate | `FlowCreated`, `TaskAttachedToFlow`, `FlowStarted`, `FlowPaused`, `FlowResumed`, `FlowCancelled`, `FlowClosed` [^12][^8] |
| AgentAggregate | `AgentRegistered`, `AgentHeartbeatReceived`, `TaskAssignedToAgent`, `AgentMarkedBusy`, `AgentMarkedIdle`, `AgentDisabled`, `AgentEnabled`, `AgentCapabilitiesUpdated` [^11][^8] |
| MachineAggregate | `MachineRegistered`, `MachineHealthReported`, `SessionOpened`, `SessionClosed`, `BrowserCapabilityAttached`, `ShellCapabilityAttached` [^11][^8] |
| PluginAggregate | `PluginProposed`, `PluginBuilt`, `PluginValidated`, `PluginInstalled`, `PluginActivated`, `PluginDisabled`, `PluginRolledBack` [^12][^9] |
| WorkflowAggregate | `WorkflowVersionProposed`, `WorkflowVersionValidated`, `WorkflowVersionApproved`, `WorkflowVersionActivated`, `WorkflowVersionRolledBack` [^8][^9] |
| InstallationAggregate | `InstallTokenIssued`, `InstallationRegistered`, `InstallationConfirmed`, `InstallationUpgraded` [^3][^4] |
| ApprovalAggregate | `ApprovalRequested`, `ApprovalGranted`, `ApprovalRejected`, `ApprovalExpired` [^8][^9] |

## Read modele

UI i automaty nie powinny czytać bezpośrednio event streamów, tylko projections. CQRS zakłada właśnie osobne modele odczytu, zoptymalizowane pod konkretne widoki i zapytania.[^13][^7]

Najważniejsze projections:

- **OperationalFeedProjection** — live feed zdarzeń dla firmy.[^11][^8]
- **TaskBoardProjection** — lista tasków, ownerzy, priorytety, SLA.[^7]
- **AgentFleetProjection** — status agentów, hostów, capability, obciążenie.[^11]
- **WorkflowVersionsProjection** — aktywne i oczekujące workflowy.[^9]
- **PluginCatalogProjection** — zainstalowane pluginy, wersje, health, rollback.[^9]
- **AuditTrailProjection** — pełna historia dla compliance i debugowania.[^8]
- **CostAndUsageProjection** — wykorzystanie modeli, koszt OpenRouter, cache hit rate.[^14]


## Stack MVP

Dla MVP w Compose nie budowałbym wszystkiego od razu na Kafka + 12 serwisach. Na początek wystarczy prosty układ z EventStoreDB, Postgres, Redis/NATS, orchestratorem, projekcjami, web UI i 2 workerami. EventStoreDB ma gotowy obraz Dockera i jest projektowane właśnie jako event-native database do systemów event-sourced.[^15][^3]

Proponowany stack:

- **eventstoredb** — source of truth dla eventów.[^3]
- **postgres** — read models / projections / reporting DB.[^2][^4]
- **redis** lub **nats** — krótkie kolejki, schedulowanie, pub/sub dla runtime. Przy MVP jeden bus wystarczy.[^16][^4]
- **orchestrator-api** — write model, command handlers, sagi, router.
- **projector** — subskrybuje eventy i buduje read modele w Postgresie.
- **web-ui** — panel zarządczy.
- **shell-agent-1**, **shell-agent-2** — przykładowi wykonawcy.
- **browser-host-agent** — poza Compose, na hoście użytkownika.[^17][^18][^3]


## Układ Compose

Minimalny `docker-compose.yml` dla MVP powinien mieć jedną sieć aplikacyjną i trwałe wolumeny dla EventStoreDB oraz Postgresa. Docker Compose jest standardowo używany do lokalnych laboratoriów EventStoreDB i Redpanda/CDC, więc taki setup jest w pełni realistyczny na etap prób.[^4][^2][^3]

Układ usług:

- `eventstoredb:2113`
- `postgres:5432`
- `redis:6379` albo `nats:4222`
- `orchestrator-api:8000`
- `projector:8001`
- `web-ui:3000`
- `shell-agent-a`
- `shell-agent-b`[^2][^3]


## Jak działa przepływ

Przepływ MVP powinien wyglądać tak: UI wysyła komendę, orchestrator zapisuje eventy do EventStoreDB, projector aktualizuje read model w Postgresie, a runtime publikuje zlecenie do agenta. Taki podział zachowuje czyste CQRS/ES i jednocześnie pozwala szybko zbudować prostą operacyjną aplikację.[^1][^10][^3]

Przykład:

1. UI wysyła `CreateTask`.[^10]
2. `TaskAggregate` emituje `TaskCreated`.[^1]
3. Projector aktualizuje `task_board` i `operational_feed` w Postgresie.[^7]
4. Saga wysyła `AssignTaskToAgent`.[^8]
5. Agent bierze zadanie i emituje runtime eventy typu `TaskExecutionStarted`, a po wyniku `TaskCompleted` lub `TaskFailed`.[^11][^8]

## Co wdrożyć najpierw

Na pierwszy sprint ograniczyłbym się do:

- 3 agregatów: `Task`, `Agent`, `Workflow`;[^10][^1]
- 1 approval gate;[^9]
- 1 worker shellowy;[^3]
- 2 projections: feed i task board;[^7]
- EventStoreDB + Postgres + Redis/NATS + orchestrator + UI.[^4][^3]

To wystarczy, żeby pokazać:

- event sourcing,
- CQRS,
- multi-agent routing,
- approval,
- i podstawę pod ewolucję workflowów.[^8][^9][^7]

Jeśli chcesz, w następnym kroku mogę przygotować **konkretny szkic `docker-compose.yml`** oraz **przykładowe JSON-y komend i eventów** dla `TaskAggregate`, `AgentAggregate` i `WorkflowAggregate`.
